# 3.0 Design Note: Pattern / Template Moves

Status: **Draft** (no code). Author: shape-only.

## Problem

A common netscoot refactor gets re-applied across several repositories in the same
organization. Today each application is a hand-authored sequence of `Move-DotnetProject`
(or `Move-PowerShellModule`, or...) calls. Errors of omission are easy: a project that
matched the spirit of the rule got left behind because nobody added it to the script.
Errors of accidental scope are easy too: a project that didn't match got moved because
the script's glob was looser than intended.

A *recipe* captures the policy once, in declarative form, and applies it deterministically.
A reviewer can read the recipe before it runs; a different repository can replay the same
recipe; an AI agent can author one from a natural-language request.

## Non-goals (3.0)

- Bidirectional / inverted recipes (apply forward + roll back by re-running the recipe
  inverted). The journal already gives us reversal of *individual moves*; reversing a
  whole recipe is reversing each move it journaled, which `Undo-Netscoot -After` already
  expresses. No new construct needed.
- A central recipe registry, catalog, or distribution mechanism. Recipes live as files
  in source control.
- Cross-repository journal sync (a recipe applied to repo A should not affect repo B's
  journal).

## Shape

A recipe is a small declarative file. It names one or more **steps**; each step is a
**selector** plus a **transform**. The recipe is engine-agnostic - the per-step engine is
determined by `Resolve-MoveEngine` on each selected file at apply time, just as
`Invoke-Netscoot` does today.

```powershell
# Example: 3.0 pattern.psd1 (illustrative; final format TBD - see Open Questions)
@{
    SchemaVersion = '1'
    Description   = 'Promote test projects out of src/ into tests/.'
    Steps         = @(
        @{
            Selector    = @{ Path = 'src/**'; Match = '*.Tests.csproj' }
            Destination = 'tests/${BaseName}'
        }
        @{
            Selector    = @{ Path = 'src/Shared/**'; Match = '*.props' }
            Destination = 'build/'
        }
    )
}
```

The selector names what to act on in repo-relative terms. The transform names where it
goes, with variable substitution from the selected file's properties (`${BaseName}`,
`${ProjectName}`, `${ParentFolder}`).

## Lifecycle

```
Save-NetscootRecipe → recipe.psd1   (capture or hand-author)
Test-NetscootRecipe -Path recipe.psd1 -RepositoryRoot R    (dry-run; lists steps)
Invoke-NetscootRecipe -Path recipe.psd1 -RepositoryRoot R   (apply, journaled per step)
```

`Test-` is the read-only sibling of `Invoke-`. It expands every selector against the
current repo and prints the resulting move plan in `-WhatIf -Verbose` shape: each step's
source list, destination, and the engine that would handle it. No changes.

`Invoke-` walks the same plan and calls the appropriate mover per file - existing movers,
unchanged. Each individual move is journaled normally. If the run is interrupted,
`Repair-NetscootJournal` recovers the partial state exactly as it does for ad-hoc moves
today.

## Idempotence

Applying a recipe twice must be a no-op (the second run sees the source missing and the
destination present at every step). The expansion of the selector against the current
repo naturally yields zero matches the second time, so the recipe's behavior already
expresses "do nothing if already applied" without an explicit idempotence flag.

The contract: **selectors are evaluated against the current repo state, not the recipe's
intent.** A selector that read `src/**` originally is still `src/**` on the second run;
if `src/` is now empty of matches, the step is a no-op.

## Cross-repo portability

Recipes use repo-relative paths. They do not contain absolute paths. A recipe authored
in repo A applies cleanly to repo B if (and only if) repo B has files that match the
selectors. When it doesn't, the affected step matches zero files and is a no-op - not an
error. The recipe-level result reports per-step match counts so the operator can see
what landed where.

## What survives repo schema change

- **Selector that matches nothing**: no-op (see Idempotence). The recipe completes with
  that step's match count = 0, which is surfaced in the result.
- **Selector that matches more than expected** (e.g. a new subfolder appeared under
  `src/Shared/`): every match is moved per the transform. This is intentional - a recipe
  is a *policy*, not a one-time picklist. The remediation is to tighten the selector,
  not to special-case the recipe.
- **Recipe schema mismatch** (recipe authored under 4.0, run under 3.0): the parser
  refuses to apply with a clear `RecipeSchemaUnsupported` error and prints the
  recipe's `SchemaVersion`. 3.0 ships with `SchemaVersion = '1'`.

## Composability

Each step delegates to the existing per-engine mover. The move is journaled normally,
which means:

- `Undo-Netscoot -After <recipe-start-time>` reverses the entire recipe.
- `Undo-Netscoot -Id <step-id>` reverses a single step. The journal's per-move ids are
  unchanged.
- `Repair-NetscootJournal` recovers from a mid-recipe interruption exactly as it does for
  ad-hoc moves.

Recipes do not introduce a new journal type. The journal entries from a recipe run are
indistinguishable from hand-authored moves at the journal layer; the recipe's identity
lives in a recipe-run object that pairs the recipe path with the journal-id list, kept
for the run's duration only.

## Authoring

Two paths:

### Hand-authored

Write the `.psd1` file. The shape is small enough that an AI agent can produce it from a
natural-language description ("move every project under src/ whose name ends in `.Tests`
into tests/"), and a human can review it before applying.

### Captured from journal

`Save-NetscootRecipe -FromJournal -Since <time> -RecipeFile X` reads the per-user move
journal between two timestamps and projects each move into a parameterized recipe step,
generalizing the per-instance paths into selectors where possible. This is the killer
feature for cross-repo replay: do the refactor once in repo A with the hand-driven
movers, save it as a recipe, then run it against repos B-Z.

The generalization is heuristic and can fail to produce the user's intent. The output is
always a draft for review, never auto-applied. The captured recipe's `Description` field
quotes the source journal range so the provenance is obvious.

## Cmdlet surface (proposed)

| Cmdlet                 | What it does                                                              |
| :--------------------- | :------------------------------------------------------------------------ |
| `Save-NetscootRecipe`  | Emit a recipe file (hand-authored from selectors, or captured from journal). |
| `Test-NetscootRecipe`  | Dry-run: expand selectors against the current repo, list the move plan.    |
| `Invoke-NetscootRecipe`| Apply: run each step through the appropriate mover, journaled per move.    |

Output types (proposed):

- `Netscoot.Recipe` - the parsed recipe (steps, schema version, description).
- `Netscoot.RecipeStep` - one step's plan (selector, expansion, destination, engine).
- `Netscoot.RecipeResult` - one step's outcome (matches found, moves performed, errors).

The journal-side state, plan rendering, and -WhatIf semantics are inherited from the
existing movers; no new structured-error types are needed.

## Open questions

1. **Recipe file format.** `.psd1` (native to PowerShell, matches the project's existing
   data files like `docs/output-types.psd1`) vs `.yaml` (portable across tools, friendlier
   to operators not on PS, but adds a YAML parser dependency) vs `.json`. The conservative
   choice is `.psd1` for 3.0 and accept a richer format later if there's demand.
2. **Selector grammar.** Glob-only (`src/**/*.Tests.csproj`)? Or glob + predicate (e.g.
   "matches the glob *and* the .csproj contains `<IsTestProject>true</IsTestProject>`")?
   The glob-only form covers most cases and is reviewable at a glance; predicates are an
   extension point if needed.
3. **Variable substitution scope.** `${BaseName}`, `${ProjectName}`, `${ParentFolder}` are
   obvious. Should we also expose `${RepositoryRoot}`-relative computations? Probably not
   in v1; keep substitution lexical.
4. **Partial failures.** If step 3 of 5 fails mid-recipe, should the recipe runner abort
   and stop, or continue with steps 4-5? Default: stop on the first per-step error;
   `-ContinueOnError` to opt into best-effort. (The journal still records steps 1-2 so
   they can be reversed independently.)
5. **Cross-engine recipes.** A recipe that mixes `.csproj` moves and `.ps1` moves in the
   same file - is that allowed? Tentatively yes (the engine is per-step from
   `Resolve-MoveEngine`), but reviewers will read mixed recipes as a code smell. No
   special validation; just a documentation note.
6. **Capture-from-journal generalization.** How aggressive is the generalization? Two
   moves of `src/A/A.csproj` → `libs/A/A.csproj` and `src/B/B.csproj` → `libs/B/B.csproj`
   could become `src/${ProjectName}/** → libs/${ProjectName}/`. But if the journal also
   has `src/Special/Special.csproj` → `vendor/Special/Special.csproj`, the captured recipe
   should *not* fold all three. The heuristic likely needs a confidence floor and the
   captured-recipe Description should flag any moves the heuristic dropped.
7. **Naming.** `*-NetscootRecipe`? `*-MovePattern`? `*-Restructure`? The chosen verb-noun
   has to coexist with the existing `Move-*` surface; "Recipe" reads natural and is
   distinct.

## What this does not change

- The per-mover contracts (`Move-DotnetProject` etc.) - recipes call them, they are
  unchanged.
- The journal format - recipe runs produce ordinary journal entries.
- The error / output / ShouldProcess shape of any existing cmdlet.
- Branch protection, release flow, CI gates.

## Why now (rather than 4.0)

Pattern moves change the unit of work from "one move" to "one policy", which is the
right altitude for cross-repo work and AI-agent authoring. Doing it as a 3.0 feature means
the recipe surface ships under the existing journal/rollback/repair guarantees, which is
what makes recipes safe to apply. Deferring to 4.0 means accumulating ad-hoc scripts in
the meantime that don't compose with the rest of the toolkit.
